# BleCore 项目框架入门说明

> 面向新手：从「打开 App」到「收发包」，把模块、页面、入口和数据流串起来。  
> 工程根目录：`Ble_And_Classic_BT_Helper`，工程名：**BleCore**。

---

## 1. 一句话理解这个项目

这是一个 **同时支持 BLE（低功耗蓝牙）和经典蓝牙 SPP** 的调试 App：

- **BLE**：扫描 → 连接 → 看 GATT 服务/特征 → 读写/Notify → 看接收日志
- **SPP**：选已配对设备 → 串口式 Terminal → 收发字节/协议解析

底层库拆成多个模块，Demo UI 在 `app` 里。

---

## 2. 模块结构（先看这一层）

```
BleCore
├── :app        Demo 应用（页面、按钮、跳转都在这里）
├── :ble        BLE 核心库（扫描/连接/读写/Notify）
├── :spp        经典蓝牙 SPP 终端 + 私有协议解析
└── :support    公共基类、工具（BaseActivity、Application 等）
```

| 模块 | 包名大致 | 做什么 |
|------|----------|--------|
| `app` | `com.bhm.demo` | 用户能看到的界面；依赖另外三个模块 |
| `ble` | `com.bhm.ble` | 可复用的 BLE SDK，入口是 `BleManager` |
| `spp` | `de.kai_morich.simple_bluetooth_terminal` | SPP 终端、串口服务、协议解析器 |
| `support` | `com.bhm.support.sdk` | `BaseApplication`、`BaseActivity` 等脚手架 |

依赖关系（箭头表示「谁依赖谁」）：

```
app ──► ble
app ──► spp
app ──► support
```

新手记法：

- 改 **界面/流程** → 多半在 `app` 或 `spp` 的 UI
- 改 **BLE 连接能力** → 看 `ble`
- 改 **SPP / 协议解析 / Sleep Record** → 看 `spp`

## 3. App 怎么启动（入口）

### 3.1 Application

`app/src/main/AndroidManifest.xml` 里：

```xml
android:name="com.bhm.support.sdk.common.BaseApplication"
```

也就是启动时先跑 `support` 模块的 `BaseApplication`（初始化 Context、日志等）。

### 3.2 第一个页面（Launcher）

启动 Activity：

- 类：`com.bhm.demo.ui.TransportSelectActivity`
- 布局：`activity_transport_select.xml`
- 作用：**让用户选 BLE 还是 SPP**

对应代码逻辑很简单：

| 按钮 | 跳转到 |
|------|--------|
| 进入 BLE 控制 | `com.bhm.demo.ui.MainActivity`（BLE 扫描页） |
| 进入 SPP 控制 | `de.kai_morich.simple_bluetooth_terminal.MainActivity`（SPP 容器页） |

## 4. 页面总览与跳转关系

### 4.1 总流程图（建议先背这张图）

```
TransportSelectActivity          ← App 启动入口
├── BLE 路径
│   MainActivity                 ← 扫描 / 连接
│   ├── DetailOperateActivity    ← GATT 操作 + 接收面板
│   │   └── PresetWriteCommandActivity  ← 管理预设写命令
│   └── ScanBroadcastActivity    ← 广播/扫描记录预览
│
└── SPP 路径
    spp.MainActivity             ← 只有一个 Fragment 容器
    └── DevicesFragment          ← 设备列表
         └── TerminalFragment    ← 真正收发终端（可返回上一页）
```

### 4.2 BLE 侧页面

| 页面 | 布局 | 干什么 |
|------|------|--------|
| `TransportSelectActivity` | `activity_transport_select.xml` | 选传输方式 |
| `MainActivity` | `activity_main.xml` | BLE 扫描列表、连接 |
| `DetailOperateActivity` | `activity_detail.xml` | 服务/特征操作、接收数据、Sleep Record 按钮 |
| `ScanBroadcastActivity` | `activity_scan_broadcast.xml` | 看扫描广播内容 |
| `PresetWriteCommandActivity` | `activity_preset_write_command.xml` | 预设写命令管理 |
| `OptionSettingActivity` | `activity_setting.xml` | BLE 选项（已注册，当前主流程较少用到） |

BLE 路径 **全是 Activity**，没有 Fragment。

### 4.3 SPP 侧页面

| 页面 | 布局 | 干什么 |
|------|------|--------|
| `spp.MainActivity` | `spp_activity_main.xml` | 宿主：Toolbar + `R.id.fragment` |
| `DevicesFragment` | ListFragment + `device_list_*.xml` | 已配对设备列表、选 SPP UUID |
| `TerminalFragment` | `fragment_terminal.xml` | 收发、HEX、同步保存、协议解析 |
| `SleepRecordHistoryDialogFragment` | `dialog_sleep_record_history.xml` | Sleep Record 历史小窗口 |
| `TapParameterDialogFragment` | `dialog_tap_parameters.xml` | Tap 参数配置 |

SPP 路径是：**一个 Activity + 多个 Fragment 切换**。

### 4.4 页面怎么切换（Intent / Fragment）

**BLE：Activity → Activity（Intent）**

常见参数：

| Extra Key | 含义 | 典型路径 |
|-----------|------|----------|
| `"data"` | `BleDevice` 对象 | 扫描页 → 详情页 |
| `"disConnectWhileClose"` | 关闭详情页是否断开连接 | 部分入口会传 `true` |
| `"ble_device"` | 预览广播用的设备 | → `ScanBroadcastActivity` |

**SPP：Fragment replace + 回退栈**

1. `spp.MainActivity` 启动时 `add` 一个 `DevicesFragment`
2. 点设备 → `replace` 成 `TerminalFragment`，并 `addToBackStack`
3. 按返回键 → 弹出 Terminal，回到设备列表

终端里还会把设备 MAC 通过 `arguments`（key：`"device"`）传给 `TerminalFragment`。

## 5. 界面布局怎么对应代码（新手看布局的方法）

Android 习惯：

1. `res/layout/xxx.xml` 定义控件（Button、TextView、RecyclerView…）
2. Activity/Fragment 里用 **ViewBinding** 找到控件，例如 `viewBinding.btnEnterBle`
3. 点按钮 → `startActivity` / `FragmentTransaction` / 调 ViewModel

以详情页为例（`activity_detail.xml`）：

| 区域 | 典型控件 ID | 作用 |
|------|-------------|------|
| 顶部 | `toolbar` | 标题、返回 |
| 中上 | `recyclerView` | GATT 服务/特征列表 |
| 中下 | `llReceivePanel` / `tvReceiveData` | 接收数据显示 |
| 按钮 | `btnClearReceive`、`btnSleepRecord` | 清接收区、打开 Sleep 历史 |
| 底部可折叠 | `llContent` / `logRecyclerView` | 操作日志 |

SPP 终端（`fragment_terminal.xml`）：

| 区域 | 典型控件 ID | 作用 |
|------|-------------|------|
| 上 | 文件名、保存 Checkbox | 同步落盘 |
| 中 | `receive_text` + `receive_scroll` | 接收区 |
| 下 | `send_text`、发送按钮 | 发送区 |

## 6. BLE 数据流（从空中到屏幕）

```
设备 Notify
   → BleManager（:ble）
   → DetailViewModel.appendReceiveData
   → BleReceiveFormatter.format（拆私有协议帧）
   → receiveChunkFlow
   → DetailOperateActivity 追加到 tvReceiveData
```

要点：

- **连接/Notify 能力**在 `:ble` 的 `BleManager`
- **格式化/协议解析**在 `app` 的 `BleReceiveFormatter`（内部复用 `:spp` 里的解码器）
- 若匹配 Sleep 上报（见第 8 节），会同时写入 `SleepRecordHistoryStore`

---

## 7. SPP 数据流（从串口到屏幕）

```
RFCOMM 读到字节
   → SerialSocket
   → SerialService（可在后台继续跑）
   → TerminalFragment.receive
   → PrivateProtocolStreamDecoder 拼完整帧
   → 按 subId 解析（IMU / Touch / Sleep…）
   → 追加到 receive_text
```

要点：

- UI 是 `TerminalFragment`
- 真正连蓝牙串口的是 `SerialService` + `SerialSocket`
- 协议解析类都在 `:spp` 包下

## 8. Sleep Upload / Sleep Record 历史（两边共用）

### 8.1 协议结构（概念）

设备周期性上报 `sleep_upload_data_t`（packed，当前 payload **87 字节**），其中：

- `sleep_record_len`：有效字节长度，合法值 `0/4/8/12/16/20`
- `sleep_record[20]`：每 4 字节一组（`'D' + stage + posture + '\n'`）

### 8.2 识别条件（BLE / SPP 一致）

- group = `0xFF`
- subId = `0x08`
- payload 长度 = `SleepUploadDataParser.PAYLOAD_SIZE`（87）

### 8.3 相关文件

| 文件 | 作用 |
|------|------|
| `SleepUploadDataParser.kt` | 解析结构体、格式化文本 |
| `SleepRecordHistoryStore.kt` | 内存里攒「有效 sleep_record」历史 |
| `SleepRecordHistoryDialogFragment.kt` | 小窗口 UI |
| `dialog_sleep_record_history.xml` | `ScrollView` + `TextView` + Clear |

### 8.4 入口

| 通道 | 怎么打开 |
|------|----------|
| BLE | 详情页 `Sleep Record` 按钮 |
| SPP | Terminal 菜单 `Sleep Record` |

说明：

- 主接收区仍会刷 1Hz 完整上报
- 历史窗口只追加「有效 sleep_record」，避免被冲掉
- `Clear` 只清历史，不清主接收日志
- 数据在进程内存中，杀进程后清空

### 8.5 小窗口里 ScrollView / TextView 干嘛

- **TextView**：真正放文字
- **ScrollView**：内容太长时可以滑动
